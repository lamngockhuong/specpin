Specpinはビジネス仕様をライブUI要素に貼り付ける3層システムを通じて機能します：Gitバージョン管理されたspecリポジトリ、ローカルsidecarサーバー、そしてspecをマッチしてレンダリングするブラウザextension。

## 3層フロー

```
.specs/ (リポジトリ内) -> specpin serve (Go sidecar) -> ブラウザextension
```

1. **specは`.specs/`に存在します**。プロジェクトリポジトリ内のJSONファイルとして。Gitでバージョン管理され、プルリクエストでレビューでき、他のコードと同様に差分確認ができます。

2. **`specpin serve`がローカルで公開します**。HTTP経由（localhostのみ、トークン認証、Server-Sent Eventsによるライブリロード）。sidecarは`127.0.0.1`に自動選択ポートでバインドする小さなGoバイナリです。データはマシンの外に出ません。

3. **ブラウザextensionが接続し**、specを取得し、各specをページのDOM要素にマッチさせ、レンダリングします（ツールチップ、サイドバー、またはモーダルとして）。ディスク上のspecファイルを編集すると、ページが自動的に更新されます。

## specが要素にリンクされ続ける仕組み

各specはターゲット要素に関する複数のシグナルを持つ**フィンガープリント**を保存します：

- `data-spec-id`属性（存在する場合、マッチングは完全一致）
- `data-testid`およびその他のtest-idパターン
- ARIA属性（role、label、describedby）
- 生成されていない`id`属性
- 最適化されたCSSセレクター
- XPath
- テキストコンテンツ
- ページ上の位置
- フレームワーク固有のヒント

ページを開くと、extensionはDOMを走査して各specのフィンガープリントをマッチさせようとします。まず完全なアンカー（`data-spec-id`、安定したtest-id、aria）をチェックし、次にユニークなCSSセレクターにフォールバックします。高信頼度のマッチが見つからない場合、specには**要レビュー**フラグが立てられ、手動で確認または再リンクできます。

このマルチシグナルアプローチにより、specはリファクタリングを経ても存続します。CSSクラスを変更しても`data-testid`を保持すれば、specは引き続き要素を見つけます。両方を変更してもaria-labelが同じであれば、マッチングは引き続き機能します。

### `data-spec-id`属性を追加する

重要な要素（ボタン、入力フィールド、ナビゲーション項目）については、ソースコードに`data-spec-id`属性を追加します：

```html
<button data-spec-id="submit-order">注文する</button>
```

extensionはこのspecを完全一致でマッチさせ、他に何が変更されても問題ありません。これが最も信頼性の高いアンカーです。

## マッチにレビューが必要な場合

extensionがspecを確信を持ってマッチできない場合、**要レビュー**とマークします。これは以下の場合に発生します：

- 要素が削除されたか大幅にリファクタリングされた。
- 複数の要素がフィンガープリントにマッチする（曖昧）。
- 位置や部分テキストなど弱いシグナルのみが残っている。

extensionのspecリストに黄色いバッジ付きでspecが表示されます。**specを編集**をクリックし、**要素を再リンク**をクリックして、ページ上の正しい要素をクリックします。extensionが新しいフィンガープリントをキャプチャし、`.specs/`ディレクトリに保存します。specは再びマッチします。

## ローカルファースト、プライベート

すべてのspecデータはローカルマシンのみを経由します。sidecarは`127.0.0.1`にバインドし、bearerトークンを要求します（`specpin serve`実行時に表示されます）。extensionはlocalhost HTTP経由でspecを取得し、書き込みは`.specs/`ディレクトリにpretty-printされたJSONファイルとして戻ります。外部サービスがspecやページを見ることはありません。詳細は[セキュリティとプライバシー](/ja/concepts/security-and-privacy/)を参照してください。
